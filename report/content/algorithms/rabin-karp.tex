\subsection{The Rabin-Karp Algorithm}

\subsubsection{Introduction \& Application}
\begin{itemize}
    \item \textbf{Core idea}: The Rabin-Karp algorithm uses a rolling hash function to find occurrences of a pattern $P$ within a text $T$. Instead of comparing the pattern character by character for all possible shifts, it calculates a hash value for the pattern and a hash value for each window of the text. If the hash values match, it performs a character-by-character comparison to verify if it is a true match or a hash collision. The key advantage is that the hash value of the next window can be computed in $\mathcal{O}(1)$ time by dropping the first character and appending the new character.
    \item \textbf{Real-world applications}:
    \begin{itemize}
        \item Plagiarism Detection: Comparing documents to find exact matches of sentences or paragraphs in a large database of academic papers.
        \item Bioinformatics: Searching for specific DNA sequences within massive genetic codes efficiently.
    \end{itemize}
\end{itemize}

\subsubsection{Pseudocode}
\textbf{Notation:}
\begin{itemize}
    \item \textbf{Text $T$ ($R$ rows, $C$ columns)}: The 2D grid of characters.
    \item \textbf{Pattern $P$ ($K$ keywords)}: The list of patterns to search for.
    \item \textbf{Modulo $q$}: A large prime number used to prevent integer overflow and minimize hash collisions (e.g., $q = 1000000007$).
    \item \textbf{Base $d$}: The number of characters in the alphabet (e.g., $d = 256$ for ASCII).
\end{itemize}

\textsc{Rabin-Karp-Matcher}($T, P, d, q$)
\begin{algorithmic}[1]
\State $n = \mathrm{length}[T]$
\State $m = \mathrm{length}[P]$
\State $h = d^{m-1} \bmod q$
\State $p\_hash = 0$ 
\State $t\_hash = 0$ 
\For{$i = 1$ \textbf{to} $m$}
    \State $p\_hash = (d \cdot p\_hash + P[i]) \bmod q$
    \State $t\_hash = (d \cdot t\_hash + T[i]) \bmod q$
\EndFor
\For{$s = 0$ \textbf{to} $n - m$}
    \If{$p\_hash == t\_hash$}
        \If{$P[1..m] == T[s+1 .. s+m]$}
            \State append $s$ to $Positions$
        \EndIf
    \EndIf
    \If{$s < n - m$}
        \State $t\_hash = (d \cdot (t\_hash - T[s+1] \cdot h) + T[s+m+1]) \bmod q$
        \If{$t\_hash < 0$}
            \State $t\_hash = t\_hash + q$
        \EndIf
    \EndIf
\EndFor
\end{algorithmic}

\textbf{Notation:}
\begin{itemize}
    \item \textbf{Text $T$ (length $n$)}: The main text being searched.
    \item \textbf{Pattern $P$ (length $m$)}: The string that we are looking for.
    \item \textbf{Shift ($s$)}: The current offset of the text $T$.
    \item \textbf{Positions ($[s_1, s_2, \dots]$)}: A list storing the indices where a complete match is found.

\textbf{Example:} text $T = [a, a, b, c, a, b, b]$ and pattern $P = [a, b, c]$. \\
For simplicity, we map characters to integers ($a=1, b=2, c=3$), set base $d=10$, and prime $q=13$. \\
Hash of pattern $P$: $p\_hash = (1\cdot 10^2 + 2\cdot 10 + 3) \bmod 13 = 123 \bmod 13 = \mathbf{6}$.

\subsubsection{Discussion}
\begin{itemize}
    \item In \textbf{Scenario 1} (Impact of Grid Size), where the number of patterns $K$ is small and the grid scales from $10 \times 10$ to $500 \times 500$, the Rabin-Karp algorithm is highly efficient. Due to its $\mathcal{O}(1)$ rolling hash update, the running time behaves almost linearly, similar to $\mathcal{O}(R \cdot C)$.
    
    \item In \textbf{Scenario 2} (Impact of Dictionary Size), where the grid is fixed at $100 \times 100$ and $K$ increases, our specific implementation loops through the grid separately for each pattern. Therefore, the execution time scales linearly with $K$. While this is faster than Brute-Force in terms of raw character comparisons, it does not fully exploit Rabin-Karp's multi-pattern matching potential (which could theoretically search all patterns simultaneously using a unified hash table).
    
    \item \textbf{In conclusion}, the Rabin-Karp algorithm provides a significant speedup over Brute-Force by using mathematical hashing to filter out unnecessary comparisons. It is an excellent choice when dealing with large texts, provided a robust prime number $q$ is chosen to avoid worst-case hash collision degradation.
\end{itemize}
